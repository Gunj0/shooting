# コーディング規約: Mini Shooting

このドキュメントは、リポジトリを一貫したスタイルで育てていくための規約をまとめたものです。
[設計書.md](./設計書.md) のアーキテクチャ・仕様と合わせて参照してください。

---

## 1. 言語・環境

- **Vanilla JavaScript (ES Modules)** のみ使用する。TypeScript・JSX・バンドラは導入しない。
- ブラウザ向けコードは `js/` 配下に置き、Node.js 向けテストコードは `tests/` 配下に置く。
- ブラウザ向けコードで `require()` / CommonJS は使用しない。

---

## 2. 命名規則

| 対象 | 規則 | 例 |
|------|------|-----|
| 変数・関数 | `camelCase` | `bulletSpeed`, `updateEnemies()` |
| クラス（将来使用する場合） | `PascalCase` | `GameEntity` |
| 定数（モジュールトップレベル） | `SCREAMING_SNAKE_CASE` | `GAME_WIDTH`, `SHOOT_INTERVAL` |
| ファイル名 | `camelCase.js` | `systems.js`, `constants.js` |
| CSS クラス名 | `kebab-case` | `game-area`, `hit-flash` |
| HTML 要素 ID | `camelCase` | `gameOverOverlay`, `finalScore` |

- ブール値を返す変数・引数は `is` / `has` プレフィックスを付ける（例: `isGameOver`, `hasStarted`）。
- イベントハンドラは `handle` プレフィックスを付ける（例: `handleKeyDown`, `handleCollisions`）。

---

## 3. ファイル構成と追加ルール

### 既存ファイルの役割

| ファイル | 役割 |
|----------|------|
| `js/constants.js` | 全定数の一元管理 |
| `js/state.js` | `gameState` 初期値の生成のみ |
| `js/systems.js` | ゲーム進行ロジック（state を直接変更） |
| `js/render.js` | 毎フレームの DOM 再構築 |
| `js/storage.js` | `localStorage` の読み書きのみ |
| `js/utils.js` | ゲームロジックに依存しない純粋関数（`clamp`, `isColliding` など） |
| `js/script.js` | エントリーポイント：DOM 取得・イベント登録・ゲームループ管理 |

### 追加ルール

- 数値・文字列の調整値はすべて `js/constants.js` に定数として追加する。他のファイルにマジックナンバーを書かない。
- ゲーム進行ロジックを追加する場合は `js/systems.js` に関数として追加し、`js/script.js` から呼び出す。
- 描画に関わる処理は `js/render.js` に集約する。
- 複数モジュールで共有できる副作用のない純粋関数は `js/utils.js` に追加する。
- 新しいモジュールファイルを追加する場合は、設計書の「2. 技術制約」も更新する。

---

## 4. JavaScript スタイル

- `const` を優先し、再代入が必要な場合のみ `let` を使う。`var` は使わない。
- 配列の更新は `map(...).filter(...)` のパターンで行い、破壊的な `push` / `splice` は `js/systems.js` 内の明示的なケース（スポーン処理など）に限定する。
- `state` オブジェクトを直接変更するスタイルは `js/systems.js` の中でのみ行う。他のモジュールは state を読み取り専用として扱う。
- アロー関数と通常の `function` 宣言はどちらでも可。ただし、1ファイル内でスタイルを統一する。
- セミコロンは省略しない。

---

## 5. CSS スタイル

- ゲームエンティティのスタイルはすべて `style/style.css` で管理する。
- クラス名は機能を表す `kebab-case` とする（例: `game-area`, `overlay-card`, `restart-button`）。
- JavaScript から状態を表すには `hidden` などの状態クラスを付け外しする。インラインスタイルは使わない。
- アニメーションの時間定数は `js/constants.js` の定数と合わせる（例: `PLAYER_HIT_FLASH_MS`）。

---

## 6. テスト規約

### テスト対象

- **テストする**: `js/systems.js`・`js/state.js`・`js/storage.js`・`js/utils.js` のような純粋関数または副作用が限定的な関数。
- **テストしない**: `js/render.js`・`js/script.js` のような DOM / `requestAnimationFrame` に依存するコード（Node.js 環境では動作しないため）。

### テストの書き方

- テストファイルは `tests/<モジュール名>.test.js` とする。
- テスト名は「何が・どういう条件で・どう振る舞う」を日本語で記述する（例: `"updateBullets は弾を上へ移動し、画面外の弾を削除する"`）。
- `node:test` / `node:assert/strict` を使用する。外部テストライブラリは追加しない。
- `Math.random` などのグローバルをモックする場合は、テスト終了後に必ず元に戻す。
- `localStorage` のモックは `tests/storage.test.js` の実装パターンに倣う。

## 7. 設計書の更新タイミング

以下の変更を行う場合は、`docs/設計書.md` を同時に更新する。

- 新しいゲームエンティティや仕様を追加したとき
- `GameState` の型定義（セクション 6）に変更があったとき
- 定数設計（セクション 7）に大きな変更があったとき
- ファイル構成（セクション 2）が変わったとき
- 操作仕様（セクション 4）が変わったとき
